# Verification Report: Login (POST /auth/v1/token?grant_type=password)

## Overall Status
PASSED WITH WARNINGS

## 1. Consistency with api-plan.md
### Issues Found
- No issues found.

### Warnings
- **`mapAuthError` rate-limited branch is heuristic.** §4 (Response Details) and §7 (Error Handling) of the implementation plan map `429` / "too many" to `RATE_LIMITED`, but `api-plan.md` §9 (Error Mapping reference implementation of `mapAuthError`) does **not** explicitly include a `RATE_LIMITED` branch — it only mentions `INVALID_CREDENTIALS`, `EMAIL_NOT_CONFIRMED`, `EMAIL_ALREADY_REGISTERED`, `PASSWORD_TOO_WEAK`, `UNKNOWN`. The plan in §3 (Used Types) and Krok 1.5 adds the rate-limit handling beyond the api-plan.md skeleton, which is a reasonable extension but should be explicitly noted as such in the api-plan.md (or the discrepancy resolved). Not blocking — additive only.
- **`generateStaticParams()` for the login page (Krok 3.1)** is mentioned, yet `[locale]/login/page.tsx` is gated by `auth.getUser()` redirect — making it inherently dynamic. The plan should clarify that `generateStaticParams()` returns the locale list only to satisfy the `[locale]` route convention (per `CLAUDE.md`'s i18n routing note), while the page itself remains effectively dynamic via `cookies()` access.
- **`current_consent_version` comparison strategy.** Krok 4.2 uses `<` operator on `current_consent_version` (e.g. `profile.current_consent_version < CURRENT_TOS_VERSION`). `api-plan.md` §4 (Walidacja zgód) explicitly leaves the version format as TODO (semver vs ISO date vs hash) — direct string comparison `<` works for ISO dates (`'2026-05-01'`) but is unsafe for semver (`'1.10' < '1.2'` lexicographically) or content hashes. The plan picks ISO date format (`'2026-05-01'`) which is internally consistent but should be explicitly tied back to the `api-plan.md` §4 TODO so reviewers know the format decision is being made here.
- **Single Idempotency-Key for guest migration insert.** Krok 5 inserts into `documents` from `migrateGuestAutosave`, but does not set an `Idempotency-Key`. `api-plan.md` §7.3 explicitly notes that PostgREST CRUD does not use `Idempotency-Key` (UI debounce + UPSERT suffices), so this is consistent — but the plan should briefly note that the `welderdoc_migrated_at` sentinel + `check_free_project_limit` trigger together provide the equivalent guarantee for the migration path (ack already partially covered in §5.3).

## 2. Consistency with prd.md
### Issues Found

### Warnings
- **US-002 acceptance criterion "Po pomyślnym zalogowaniu użytkownik trafia do listy swoich projektów"** is satisfied by `router.replace('/[locale]/dashboard')` (Krok 3.2). The PRD says "lista projektów", the plan says "dashboard" — verify these are the same destination (per `architecture-base.md` and existing routing conventions, the dashboard *is* the project list, but spelling that out in the plan would help reviewers).
- **US-007 acceptance criterion "Dane lokalne w localStorage są zachowane lub czyszczone po potwierdzeniu pomyślnej migracji"** — the plan correctly clears `welderdoc_autosave` after success (Krok 5.1), sets sentinel `welderdoc_migrated_at` first per `architecture-base.md` §13. PRD also says "W przypadku błędu migracji wyświetlany jest toast z informacją i możliwością ponowienia próby" — the plan covers the toast (`errors.guest_migration_failed`) but does **not** explicitly describe a manual retry mechanism for the user. It is implicit (data left in localStorage → next login attempts again), but a clearer "retry on next login or manual button" mention would close the loop for US-007.
- **US-007 "Po migracji projekt jest widoczny na liście projektów użytkownika"** — the plan inserts the document but does not explicitly note that the dashboard list will refresh. With `router.replace('/[locale]/dashboard')` after migration, Next.js will fetch the list fresh, so this is implicit. Acceptable; minor doc gap.

## 3. Internal Consistency
### Issues Found
- No issues found.

### Warnings
- **Order of guest migration vs. router.replace is correct but tightly coupled.** §5.3 places guest migration before §5.4 (post-submit redirect → Server Component layout consent re-check). If migration is slow (network roundtrip + DB INSERT), the user sees a stale UI before redirect. The plan does not specify a loading indicator during migration. Recommend adding to Krok 3.2 a `setIsSubmitting(true)` covering the full SDK + migration block.
- **`LoginForm` Server Component check (Krok 3.1) and `[locale]/layout.tsx` guard (Krok 4.2) duplicate `auth.getUser()` calls.** Both run on each request to `/login` for an authenticated user. The plan mentions whitelisting `/login` from the layout guard ("Pomijaj guard dla pewnych ścieżek (whitelist `/login`, `/register`, ...)"), so behavior is correct, but the duplicated `getUser()` logic should be consolidated or at least the whitelist mechanism (`headers().get('x-pathname')` or segment helper) should be implemented to avoid double work. The plan flags this as alternative (Option A vs B) but does not pick one — leaves a decision for the implementer.
- **`mapAuthError` heuristic for `PASSWORD_TOO_WEAK`** (Krok 1.5, default branch) overlaps with the rate-limit heuristic (`'too many'` check). The implementation plan should make explicit the order of checks: first `err.status === 429 || /too many|rate/`, then `password & characters` heuristic, then default `UNKNOWN`. Otherwise a 429 response with a message containing "password" could falsely map to `PASSWORD_TOO_WEAK`.
- **Stub for `/[locale]/consent-required` (Krok 6) is "out-of-scope for this task"** but is required as a redirect destination from Krok 4 layout guard. If the stub lands without a working `POST /api/consent` UI, the user could end up trapped on the page. The plan correctly notes this as a stub with TODO, but the verification suggests adding a defensive note: "If consent-required page does not yet exist or has no submit UI, the layout guard MUST not be enabled to avoid trapping users." This is the most fragile coupling in the plan.
- **`mapPostgrestError` is referenced in §5.3 (guest migration error handling) but is described as part of Krok 1 implementation in `errors.ts`.** The implementation steps and the data-flow section align, but Krok 1.4 only mentions implementing the function "according to §9 (P0001, 23514, 23502)" — it should explicitly call out `PROJECT_LIMIT_EXCEEDED` mapping (the only branch actually consumed by the login flow). Without this, `migrateGuestAutosave` would not behave as documented in §5.3 if the developer chose to defer that branch.

## 4. Summary
The plan is largely consistent with both `api-plan.md` and `prd.md`. It correctly implements US-002 and US-007 acceptance criteria, follows `CLAUDE.md` PR checklist for auth (locale redirect + consent re-check + `mapAuthError`), uses `getUser()` instead of `getSession()`, observes `welderdoc_migrated_at` sentinel ordering, and avoids the forbidden `error.message.includes(...)` antipattern by introducing `errors.ts`. The warnings are minor — tooling decisions (layout guard vs LocaleGuard, version format selection, ordering of heuristics in `mapAuthError`), additive features beyond the api-plan.md baseline (rate-limit branch), or documentation gaps that should be tightened before merge but do not block implementation. The plan is **safe to hand off to development** with the caveat that the implementer should resolve the small ambiguities (especially the `mapAuthError` check order, the `/consent-required` stub safety net, and the layout guard whitelist mechanism) during the first implementation pass.
