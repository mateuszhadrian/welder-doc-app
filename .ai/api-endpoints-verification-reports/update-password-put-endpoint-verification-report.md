# Verification Report: PUT /auth/v1/user (Update Password)

## Overall Status
PASSED WITH WARNINGS

## 1. Consistency with api-plan.md
### Issues Found
- **Cross-reference to non-existent `api-plan.md §8.3 (auth flow)`.** The header of the implementation plan claims it relates to `api-plan.md §8.3 (auth flow)`. Section 8.3 of `api-plan.md` is titled "Specjalny przypadek — webhook Paddle" and has nothing to do with auth. The relevant material (`auth.updateUser({ password })` snippet, `min_length = 8`, GoTrue rate-limit `token_refresh = 150 / 5 min / IP`) lives in the unnumbered subsection "Reset hasła — ustawienie nowego (US-004)" around line 562 and the "Hasło — walidacja (Supabase Auth / GoTrue)" block around line 1158. The cross-reference must be corrected so reviewers can locate the source.
- **`UNAUTHORIZED` is already in the `BusinessError` enum.** The plan repeatedly says `UNAUTHORIZED (dodać do enum, jeśli brak)` (§3 Used Types, §7 Error Handling table). `api-plan.md` §9.1 already declares `UNAUTHORIZED = 'unauthorized'` in the `// Generic` block of the enum sketch. The implementation should not re-declare it; the parenthetical "if missing" wording risks a duplicate symbol in the PR.
- **`mapAuthError` per api-plan §9.1 does not return `UNAUTHORIZED` for `Auth session missing`.** The plan's §7 error table says "`Auth session missing` → `UNAUTHORIZED`", but the canonical `mapAuthError` in `api-plan.md` §9.1 lines 1526-1541 has only four `switch` cases (`Invalid login credentials`, `Email not confirmed`, `User already registered`, `password ... characters`) and falls through to `UNKNOWN` for everything else, including `Auth session missing`. Either (a) the implementation must extend `mapAuthError` with an explicit `case 'Auth session missing'` (and `RATE_LIMITED` and `NETWORK_ERROR` paths — see next bullet), or (b) the plan must change its mapping table to reflect the current canonical behaviour. As written the two are inconsistent.
- **`RATE_LIMITED` and `NETWORK_ERROR` mapping is not in the canonical `mapAuthError`.** §7 of the implementation plan promises that `mapAuthError` returns `RATE_LIMITED` for HTTP 429 and `NETWORK_ERROR` for `TypeError: fetch failed`. Neither branch exists in the `mapAuthError` sketch in `api-plan.md` §9.1. As above, the plan must either extend the mapper or amend the table — currently it asserts behaviour that the contract does not implement.

### Warnings
- **`api-plan.md` example for the success body lists `2026-05-08T12:00:00Z` while the implementation plan lists `2026-05-09T12:00:00Z`.** Cosmetic only — both are valid ISO timestamps and the field is GoTrue-controlled. Not blocking.
- **The api-plan endpoint-data file (`update-password-put-endpoint-data.md`) §28 lists `email` and `data` as "optional" payload fields, the implementation plan correctly forbids them in this flow.** This is a stricter (safer) interpretation than the upstream contract, and the plan documents the rationale ("by nie wprowadzać niezwalidowanych przez UI ścieżek"). No issue, just flagging that the contract surface is intentionally narrowed.
- **Status-code remap table (§4) is "logical UI" status, not transport status.** §4 maps `Auth session missing → 401`, `Rate limit → 429`, etc. The plan correctly notes "Te statusy są w UI; warstwa transportowa GoTrue ma własne kody (HTTP 422 dla słabego hasła itd.)". This is consistent but easy to misread — a reviewer testing against GoTrue HTTP responses will see different numbers. Worth a one-line emphasis in the test plan.
- **Rate-limit reasoning leans on `token_refresh = 150 / 5 min / IP`, but `api-plan.md` calls this an indirect cap.** Both documents agree that `updateUser` has no dedicated GoTrue rate-limit and that `token_refresh` covers it transitively. No discrepancy, just confirming.

## 2. Consistency with prd.md
### Issues Found
No issues found.

### Warnings
- **PRD US-004 acceptance criteria are fully covered, but the plan's UX details add coverage for US-001-derived rules (min. 8 chars).** PRD US-004 lines 319-330 do not explicitly state "min. 8 znaków" — that constraint comes from US-001 line 286. The implementation plan correctly inherits it (§2 table, §6 "Walidacja siły hasła") because both flows produce the same `auth.users.encrypted_password` column. Worth flagging for PM that change-password (settings) flow is **not** an explicit PRD user story; it is an inferred UX surface implementing the same SDK call. If product wants change-password in MVP, PM should confirm scope.
- **PRD US-004 says "Po pomyślnym resecie użytkownik może zalogować się nowym hasłem."** The implementation plan redirects to `/login?reset=success` instead of auto-signing in. This satisfies the PRD literally (user *can* log in) but is a UX choice — Supabase actually leaves the user signed in after `updateUser` (the SDK refreshes cookies). Forcing a re-login is more conservative, but the plan should note explicitly whether the user is signed-out before the redirect or whether the `/login` page just shows a success toast over a still-active session. As written it is ambiguous.
- **No mention of the recovery link's TTL (PRD US-004: "Link jest ważny przez określony czas (min. 1 godzina)").** This is a separate ticket (the recovery-email send is `resetPasswordForEmail`, not `updateUser`), so it is correct that the plan does not cover it. Just noting that the full US-004 acceptance criteria are split across multiple tickets and a reviewer should not expect link-TTL coverage here.

## 3. Internal Consistency
### Issues Found
- **§9 step 4 redirect target contradicts §5 / §7 / §4.** Steps 4 and §5 / §7 say after reset success redirect to `/[locale]/login?reset=success`. §4 ("Efekty uboczne") additionally says "lub od razu `/[locale]` jeśli sesja jest już ważna". The plan does not pick a winner — implementer must guess. Pick one and remove the alternative, otherwise two PRs will end up with different redirects.
- **§9 prerequisite "errors.ts" overlaps with the password-flow plan but is not gated as a hard blocker.** Definition of Done lists `src/lib/supabase/errors.ts` existing as the first checkbox, but §9 step 1 calls it a "Prerequisite (TODO w osobnym lub łączonym tickecie)". A "prerequisite" that may or may not be in the same ticket is ambiguous — either it must merge first (in which case mark it blocking and create the dependency) or this ticket creates it (in which case promote step 1 out of the prerequisite section). As written a developer can plausibly start work without the file existing.
- **§5 data-flow says GoTrue rotates refresh tokens, then §6 says "inne aktywne sesje użytkownika zostaną wylogowane przy najbliższym refresh".** The first claim ("GoTrue invaliduje stary refresh token i zwraca nowy") suggests a one-token rotation; the second claim describes broad cross-device logout. These coexist in the GoTrue implementation only because of the `password_change_revoke = true` config (or equivalent), which is **not** mentioned anywhere in the plan. Either confirm the Supabase setting or relax the §6 claim.
- **§7 error table cross-references `BusinessError.UNAUTHORIZED` but the column header in §3 says "do dodania, jeśli brak".** Same inconsistency as §1: the enum already has it. Once the plan is fixed in §3, §7 should drop the parenthetical.
- **§9 step 5 (unit tests) does not mock `useTranslations`.** The test plan checks `getByText(/zbyt krótkie/i)` and i18n keys, but the form uses `useTranslations` (`next-intl`). Vitest tests must either provide a `NextIntlClientProvider` wrapper with the messages JSON or mock `useTranslations` to return identity. The current step description leaves this implicit and the tests would fail to render without setup.
- **§6 "Walidacja siły hasła" point 3 (TODO operacyjne) is mirrored in §9 step 9.** Both say "verify min_length = 8 in Supabase Dashboard". This is fine but step 9 is post-deploy and §6 is pre-implementation — pick one moment to enforce it (best: pre-implementation as part of the prerequisite, because if min_length is 6 the unit test mocking `Password should be at least 6 characters` will pass while real GoTrue accepts a 6-char password — silent contract drift).

### Warnings
- **§3 lists `UpdatePasswordFormViewModel` as a local type, not exported.** Good. But §9 step 2 i18n adds key `errors.password_mismatch`, used in §7 example. The ViewModel name and key naming are consistent — just confirm `errors.password_mismatch` is added to BOTH `pl.json` and `en.json` (the step says "obie locale" but only enumerates 6 keys; `password_mismatch` is the 7th and easy to miss).
- **§7 example code uses `router.replace('/login?expired=1')` without locale prefix.** The rest of the plan correctly says `/[locale]/login?expired=1`. The example is shorthand but a copy-paste implementer would produce a locale-broken URL. Either prefix it or add a comment.
- **Definition of Done lists `pnpm verify:routes`.** The `verify:routes` script (per CLAUDE.md) checks `vercel.json` cron paths and Paddle/consent/export handlers — none of which this ticket touches. Running it is harmless but not load-bearing for this ticket; clarify it is a "nothing-broke" smoke check, not a coverage check for this endpoint.
- **§8 Performance — "bottleneck operacyjny" suggests `disabled` button on submit.** Good UX guidance, but the unit test plan (§9 step 5) does not assert button-disabled state during pending submit. Add a fifth test case if debounce is contractual.
- **§5 mentions LocaleGuard / consent re-check after password change, but the layout-level guards are themselves not yet implemented (they are part of US-002 sign-in ticket, per CLAUDE.md PR checklist).** The plan correctly notes the guards "powinny zadziałać" but does not gate this ticket on the guards existing. If this ticket merges before the LocaleGuard ticket, post-reset users on a wrong-locale URL will not be redirected. Worth an explicit dependency note.
- **§9 step 6 (Playwright e2e) is marked "opcjonalne".** That is consistent with CLAUDE.md ("Playwright `chromium-desktop` is the only mandatory CI project; …") for new flows that lack production gating, but `e2e/reset-password.spec.ts` is an obvious candidate for the mandatory project once the recovery-email mock is wired. Worth confirming with QA.
- **Implementation plan does not mention Vercel BotID** which `api-plan.md` line 1329 marks as "TODO — rozważyć przed publicznym launchem (lekkie, brak dodatkowego sekretu)" specifically for "sign-up / reset password forms". Out of scope for this ticket but worth a one-line forward-reference.

## 4. Summary
The implementation plan is structurally sound and correctly identifies that PUT /auth/v1/user is an SDK-only flow rather than a custom Route Handler — that decision is consistent with `api-plan.md` and `tech-stack.md` §7. The two PRD acceptance criteria that fall inside this ticket's scope (US-004 "po kliknięciu linku użytkownik może ustawić nowe hasło", US-001 "min. 8 znaków") are addressed. However, the plan asserts error-mapping behaviour (`UNAUTHORIZED`, `RATE_LIMITED`, `NETWORK_ERROR` cases on `mapAuthError`) that is **not** present in the canonical `api-plan.md` §9.1 sketch, and it adds `UNAUTHORIZED` to the enum even though the symbol already exists — these are blocking inconsistencies that a developer would otherwise discover only at PR review. The §9.3 cross-reference to "§8.3 auth flow" is also wrong (§8.3 is about Paddle webhooks). The plan is safe to hand off to development **after** (1) the four §1 issues above are reconciled — extend `mapAuthError` with the missing branches OR amend the §7 mapping table to match the existing mapper; (2) the redirect ambiguity in §9 step 4 is resolved; (3) the `errors.ts` prerequisite is either promoted into this ticket or marked as a hard merge-blocker.
