# Verification Report: POST /rest/v1/documents (Create Document, US-008)

## Overall Status

PASSED WITH WARNINGS

## 1. Consistency with api-plan.md

### Issues Found

- **Status code mapping rozbieżny z resztą api-plan.md w przypadku „Nazwa pusta / >100 znaków" oraz „RLS reject".** Sekcja 4 implementation-plan twierdzi że pusty/długi `name` zwraca HTTP 400 (zgodnie z obserwowanym zachowaniem PostgREST przy `code: '23514'`), ale tabela "Status code mapping" w sekcji 4 i tabela w sekcji 7 podają `403` dla RLS reject — w api-plan.md §2.2 (linie 646-683) RLS reject po stronie PostgREST przy braku `email_confirmed_at` zwraca pusty result-set (200 z `[]` przy `select` lub `42501` permission error przy mutacji), nie standardowy `403`. Implementation-plan nie wskazuje który PostgREST kod błędu (`42501` `permission_denied` czy `PGRST...`) ma być mapowany na `UNAUTHORIZED`/`EMAIL_NOT_CONFIRMED` — `mapPostgrestError` w api-plan.md §9 (linie 1486-1524) nie pokrywa `42501`. To sprzeczność: kontrakt mappera nie wspiera ścieżki, której plan oczekuje.
- **Schemat `data` w api-plan.md §4 wymaga klucza `'shapes'`/`'weldUnits'` (`CHECK (data ? 'schemaVersion' AND ...)`), ale implementation-plan w sekcji 2 wymienia tylko `schemaVersion` w obowiązkowych kluczach jako pierwsze, a w preflighcie sprawdza `Array.isArray(data?.shapes)` / `Array.isArray(data?.weldUnits)` — to jest spójne, jednak api-plan §4 dodaje warunek `data ? 'schemaVersion'`, który mógłby być spełniony nawet gdy wartość to nie-number. Implementation-plan nakłada ostrzejszy warunek (`typeof === 'number'`), co jest w porządku, ale powinien wprost odnotować że jest to silniejszy guard niż DB CHECK (preflight UX, nie defense-in-depth).

### Warnings

- **API plan §2.2 (linia 683) wciąż dokumentuje stary, kruchy wzorzec `error.message.includes('project_limit_exceeded')`** — implementation-plan słusznie zastępuje go `mapPostgrestError`/`BusinessError`, co jest zgodne z §9 (Error Mapping), ale plan deklaruje to jako rekomendowane podejście („helper serwisu nie używa includes(...)"). Zgodne kierunkowo z §9, ale rozbieżne z literalną sekcją §2.2 — warto dodać uwagę w implementation-plan, że §9 jest źródłem prawdy.
- **Brak wzmianki o `Idempotency-Key`.** API plan §7 (linia 1401) explicitly stwierdza, że operacje SDK na `documents` nie wymagają osobnej idempotencji (UI guard wystarczy). Implementation-plan to rzeczywiście implementuje (`pending` w `useCreateDocument`), ale nie odsyła do §7 ani nie dokumentuje tej decyzji — czytelnik bez kontekstu mógłby uznać brak `Idempotency-Key` za przeoczenie.
- **Sekcja 6 punkt 6 (CSRF / cookie hardening)** wymienia `updateSession()` w `src/proxy.ts`, ale matcher proxy w MVP wyklucza `/api/*` (CLAUDE.md). Ten endpoint to PostgREST `/rest/v1/documents`, więc cookies są refreshowane przez `auth.getUser()` w helperze — fakt, ale formułowanie "updateSession() w proxy.ts" sugeruje że odświeżenie zachodzi przez proxy, co dla tej ścieżki nie jest prawdą. Drobne, ale mylące.
- **Tabela status codes w §4** podaje `400` dla walidacji DB i `500` dla `project_limit_exceeded` (P0001) — to jest zgodne z PostgREST domyślnym mapowaniem, ale api-plan.md nie potwierdza ekspliсytnie tego mapowania (§2.2 pokazuje tylko payload błędu). Ostrzeżenie nie blokujące — fakty są zweryfikowalne empirycznie, a `BusinessError` chroni klienta przed zależnością od HTTP code.

## 2. Consistency with prd.md

### Issues Found

- **US-008 AC4 wymaga „komunikat o limicie z CTA do upgrade".** Implementation-plan to realizuje (`toast.error + showUpgradeModal()`), ale wskazuje jedynie istnienie `showUpgradeModal()` bez referencji jak ma wyglądać komunikat — PRD US-046 (linia 996+) i US-047 (linia 1009+) wprost wymagają osobnego komunikatu „przekroczenie limitu projektów" (tab. status w PRD §3.7 row 1009). Implementation-plan korzysta z generic `errors.project_limit_exceeded`, co jest spójne z API §9, ale nie odnotowuje że PRD US-047 wymaga osobnego klucza i CTA.

### Warnings

- **US-007 AC5 (PRD linia 363+) wymaga że migracja jest „bez wymagania akcji ze strony użytkownika".** Implementation-plan §5 Data Flow / Wariant US-007 to spełnia (auto-trigger przez LocaleGuard / sign-in callback), ale w sekcji "Krok 4b" znajduje się fragment kodu używający `await createDocument(...)` w komponencie React. Nie jest jasne czy LocaleGuard to Server Component (gdzie `localStorage` nie istnieje) czy Client Component (`'use client'`). PRD nie precyzuje, ale architecture-base.md §17 traktuje LocaleGuard jako miejsce po `auth.getUser()`. Implementation-plan powinien jednoznacznie wskazać że migracja zachodzi po stronie klienta (po hydratacji), żeby uniknąć implementacyjnego błędu „localStorage is not defined" w SSR.
- **US-005 (PRD linia 334+) — limit 3 elementów Guest.** Implementation-plan §6.4 stwierdza poprawnie że guard jest wyłącznie client-side w `ShapesSlice`. Brak ostrzeżenia że Free user (zalogowany) **też** podlega temu limitowi 3 elementów (PRD tab §3.1 linia 46: `Free | max 3 | 1`). Implementation-plan w "Kroku 5" pokrywa to (`isGuest || isFree`), ale wymaga to szczegółowego czytania — warto byłoby uwypuklić w sekcji 6.
- **US-012 AC3 (PRD linia 448) — „Kopia jest niezależna od oryginału".** Implementation-plan w "Kroku 4c" wykonuje shallow copy (`source.data as CanvasDocument`). Jeśli store kiedykolwiek przekaże referencję do JSONB obiektu zwróconego z bazy, kolejne mutacje na kopii mogą wpłynąć na cache oryginału w pamięci klienta. W praktyce Supabase serializuje JSON, więc to nie jest realnym problemem, ale brak wzmianki o tym wymaga założenia o granicach serializacji.

## 3. Internal Consistency

### Issues Found

- **Sprzeczność między §4 (status code mapping) a §7 (error handling).** §4 podaje `403 — RLS odrzuca`, §7 podaje że RLS reject mapuje się na `UNAUTHORIZED` lub `EMAIL_NOT_CONFIRMED`, ale `mapPostgrestError` w api-plan.md §9 nie obsługuje kodu `42501` (`permission_denied`) ani PGRST116 (`Result contains 0 rows` — tu nie dotyczy, ale RLS w PostgREST mutacjach typowo zwraca `42501`). Ścieżka `RLS reject → BusinessError.UNAUTHORIZED` nie jest pokryta przez kontrakt mappera. To wymaga dodania case'u w `errors.ts` (Krok 1) — implementation-plan tego nie żąda. Krytyczne dla scenariusza email-not-confirmed, kluczowego w US-001.
- **`createDocument` zwraca `data: DocumentDto | null` z `.select('id, name, schema_version, data, created_at, updated_at').single()`,** ale `Tables<'documents'>['data']` jest typu `Json` (per uwaga w §3 i komentarz `data: command.data as unknown as Json`). W kodzie helpera (Krok 2) zwracane `data.data as unknown as CanvasDocument` jest niesprawdzonym rzutem — nie ma walidacji że odpowiedź jest faktycznie zgodna z `CanvasDocument` (np. że RPC trigger nie nadpisał `data` na coś nieoczekiwanego). To nie jest sprzeczność z planem, ale z deklarowaną w §3 zasadą „TypeScript wymaga jawnego rzutowania" — jeśli używamy rzutu, powinien być co najmniej runtime-guard (`if (!isCanvasDocument(data.data)) return error UNKNOWN`). Nie jest to wymagane przez plan, ale brakuje mu spójności z preflight-walidacją (preflight idzie do bazy ostro, response wraca surowo).

### Warnings

- **Krok 2 helper-serwis a Krok 3 hook** mieszczą się w jednym pliku `src/lib/supabase/documents.ts` ("lub osobnym ..."). Konwencja CLAUDE.md (Zustand store conventions, "Custom hook per slice") preferuje izolację hooków od czystych helperów. Implementation-plan nie egzekwuje wyboru — zdecydowanie powinien wskazać że `useCreateDocument` to client-only (`'use client'`), a `createDocument` to izomorficzny helper (zgodnie z Krokiem 2 nota). Zostawienie tego jako "lub" jest niespójne ze sztywnym podziałem CLAUDE.md.
- **Krok 7a (testy)** wymienia coverage thresholds (80/80/70/80). CLAUDE.md potwierdza te progi dla `src/lib/**`. OK, ale Krok 7b żąda testów dla `src/lib/supabase/errors.test.ts` — jest to test pliku **jeszcze nie istniejącego** (TODO Krok 1). Ostrzeżenie: w PR-checkliście (Krok 8) brak weryfikacji że Krok 1 jest faktycznie ukończony przed Krokiem 7. Mała dyscyplina kolejności.
- **Krok 4c (duplikat) używa `await supabase.from('documents').select('*').eq('id', sourceId).single()`** zamiast helpera. Brak spójności z deklaracją w §1 że "implementacja powinna być opakowana w cienki helper-serwis ... zamiast surowych wywołań SDK rozsianych po komponentach". Jeśli US-012 jest pełnoprawnym use-case'em, powinien istnieć osobny helper `getDocumentById(supabase, id)` (poza zakresem tego endpointa, ale wskazany jako implementowany razem). Rekomendacja: explicitly oznaczyć że Krok 4c zakłada istnienie GET helpera (poza zakresem), albo dopuścić surowe wywołanie tylko jako tymczasowe.
- **§7 Error Handling — wiersz „Network error / timeout"** mapuje na `NETWORK_ERROR`, ale `mapPostgrestError` w api-plan.md §9 nie zawiera tego case'u (mapuje tylko `PostgrestError`, nie wyjątki `fetch`). Plan w §7 wymaga tej mapy — Krok 1 (przepisz `errors.ts` 1:1 z §9) nie pokrywa `NETWORK_ERROR`. Trzeba zdecydować: albo dodać try/catch w helperze i ręczny mapping, albo rozszerzyć `mapPostgrestError` (sprzeczne z 1:1).

## 4. Summary

Implementation-plan jest dobrze ustrukturyzowany i pokrywa większość wymagań US-008, US-007, US-012, ale ma kilka realnych luk w warstwie error-mapping: `mapPostgrestError` zgodnie z api-plan.md §9 nie pokrywa kodu `42501` (RLS reject) ani wyjątków sieciowych, a implementation-plan polega na obu. Ścieżka „email niepotwierdzony → friendly toast + redirect na `/email-verify-required`" nie ma deterministycznego mapowania — to blokujące dla US-001/US-008 acceptance criterion „Free user nie może zapisać projektu bez potwierdzonego emaila". Pozostałe rozbieżności to ostrzeżenia (mieszanie helpera i raw SDK w Kroku 4c, niejednoznaczna kolejność Krok 1 vs Krok 7, nieoczywista granica izomorficzny/client). Plan może zostać przekazany dev-teamowi po dopisaniu w Kroku 1 obsługi PostgREST `42501` + wyjątku `fetch` (`NETWORK_ERROR`) oraz rozstrzygnięciu organizacji `useCreateDocument` vs `createDocument` (osobny plik vs jeden plik).
